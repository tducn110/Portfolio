# Featured Project 1

## Finance Tracker V3

### Short Description

A budget-first personal finance platform for managing wallets, budgets, recurring bills, savings goals, and spending analytics.

### Problem

Many expense trackers only record what users already spent. They help users look back, but they do not always help users decide what is safe to spend today.

The problem was to build a finance app that focuses not only on tracking transactions, but also on planning money before spending it.

### Solution

Finance Tracker V3 was designed around a Budget-First mindset. The app puts budgets at the center and connects them with wallets, transactions, recurring bills, savings goals, and analytics.

Instead of being only a transaction list, the app becomes a system that helps users understand their financial situation and make better spending decisions.

### My Role

Team Lead
Backend
Database
Architecture

I worked on the system structure, backend logic, database design, API flow, and core finance features. I also helped organize the project direction so the app could grow beyond a simple expense tracker.

### What I Built / Contributed

Multi-wallet financial structure
Budget-first logic
Database schema planning
Backend API structure
Transaction and wallet relationships
Savings goals logic
Recurring bills logic
AI Quick Add flow
Project architecture and technical direction

### Technical Decisions

Used TypeScript to make the app safer and easier to maintain.
Used PostgreSQL/Supabase for relational financial data.
Used Drizzle ORM for database schema and queries.
Used Zod for validation.
Used Hono API for backend routes.
Used a monorepo structure to separate web, API, database, shared schemas, and related packages.
Used AI Quick Add to convert natural language into transaction input.

### Challenge

The hardest part was understanding how finance data connects together.

A transaction is not just a row in a table. It can affect wallet balance, budget progress, analytics, savings goals, and safe-to-spend calculations. This pushed me to think more carefully about data relationships and product rules.

### What I Learned

This project taught me how to think more like a system builder.

I learned that a full-stack product needs more than a beautiful dashboard. It needs schema design, validation, API contracts, state management, clear logic, and a structure that can grow.

### Stack

# Featured Project 2

## Unsaid Words, Shared Hearts

### Short Description

An anonymous note wall for emotional support, comfort replies, and supporter moderation.

### Problem

Some people have thoughts they want to express but cannot easily say out loud.

The challenge was to create a quiet, anonymous, and safe digital space where people could write notes, read others’ stories, and send comfort without exposing their identity.

### Solution

Unsaid Words, Shared Hearts uses a simple and calm flow:

Users write an anonymous note.
Supporters review it.
Approved notes appear on the public wall.
Visitors can send comfort replies.
Replies are moderated before becoming public.

This keeps the space open enough for emotional expression, but controlled enough to protect the community.

### My Role

Creator
Product flow designer
Frontend developer
Supabase data layer builder
Moderation flow designer

I built the main experience around anonymous note submission, public wall browsing, comfort replies, and supporter moderation.

### What I Built / Contributed

Anonymous note submission
Published note wall
Comfort reply flow
Supporter moderation dashboard
Public/private content states
Supabase Postgres schema
RPC-driven data layer
Frontend routing and page structure
Testing for important flows
PWA/install direction

### Technical Decisions

Used Supabase Postgres to store notes, replies, and moderation states.
Used RPC functions to control database actions.
Used moderation status fields to decide what becomes public.
Used React Router for public routes.
Used a supporter dashboard to approve, hide, and review content.
Simplified the product scope to keep the MVP focused and safer.

### Challenge

The hardest part was balancing openness and safety.

Anonymous spaces can become risky if every message appears instantly. That is why moderation became a core part of the product, not just an admin feature.

The product also needed a gentle tone. The words, colors, interactions, and empty states had to feel calm instead of cold.

### What I Learned

This project taught me that software can be emotional.

The best solution was not to add more features, but to remove unnecessary flows and protect the core experience: write, read, comfort, moderate.

I learned how to simplify a messy idea into a focused MVP.

## How I Solve Problems

I do not start with code immediately.

Before building, I try to understand the problem behind the request.

Who is this for?
What should users understand?
What action should they take?
What features are actually necessary?
What can be removed to make the product clearer?

After that, I turn the idea into structure: pages, components, data, API, validation, and deployment.

### Step 1 — Understand the Problem

I start by asking what the product is trying to solve.

A website is not only a screen. It has to answer a real need: presenting a person, explaining an event, managing data, supporting users, or turning a story into something clear.

### Step 2 — Shape the Flow

I break the idea into user flows.

What does the user see first?
What do they click?
What data do they create?
What should happen next?
Where can confusion happen?

This helps me design products that feel simple instead of random.

### Step 3 — Build the System

I build the interface and connect it with the logic behind it.

That can include frontend components, database tables, API routes, validation, authentication, moderation states, or deployment setup.

### Step 4 — Refactor and Improve

My first version is not always perfect, and I do not pretend it is.

After the product works, I review the structure, remove unused code, improve reusable components, and make the system easier to maintain.

### Step 5 — Use AI Carefully

I use AI as part of my workflow to research faster, compare solutions, generate alternatives, review code, and refactor messy parts.

But I do not want to blindly copy code. My goal is to understand the system, make my own decisions, and use AI as a tool to build better products.
